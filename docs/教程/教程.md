# Codex Desktop worktree 截图教程

这篇教程面向已经会基础 Git、但还不熟悉 Codex Desktop worktree 工作流的用户。你会看到一条完整主线：先创建工作树，再在工作树里继续对话，把结果切到本地验证，最后合并回目标分支并删除工作树。

默认前提：

- 你已经有一个本地 Git 仓库
- 你知道 `branch`、`stash`、`commit`、`merge` 的基本含义
- 你当前使用的是和截图相近版本的 Codex Desktop

## 先看完整流程

如果你第一次接触 Codex Desktop 的 worktree，可以先记住整条流程只有这 7 步：

1. 先看懂界面里“当前工作区状态”和“当前分支”这两个位置
2. 创建新的工作树
3. 在工作树里继续对话，并给当前改动创建工作分支
4. 把工作树结果切到本地工作区验证
5. 根据验证结果，决定继续在本地修改，还是把会话移回工作树
6. 选择一种方式把改动合并回目标分支
7. 删除不再需要的工作树

你可以先把它理解成一句话：

`先在工作树里做，再切到本地看，确认没问题后再合回主分支。`

## 你可以按需阅读

如果你不是第一次看这篇教程，可以直接跳到自己关心的部分：

- 只想知道界面里哪里能看当前状态：看第 1 节
- 只想知道怎么创建工作树：看第 2 节
- 只想知道怎么把结果切到本地：看第 4 节
- 只想知道“移动到本地”和“移至工作树”怎么选：看第 5 节
- 只想知道怎么把改动合回 `main`：看第 6 节
- 只想知道怎么删除工作树：看第 7 节

## 1. 先认识界面里的关键信息

**这一步要做什么：** 先看懂界面里最重要的两个位置。

**怎么操作：** 不需要点击，先观察底部状态栏。

**你会看到什么：**

- 左下角可以看出当前会话是在本地工作区还是工作树
- 右下角可以看到当前会话所在的 Git 分支

![界面中的当前工作区状态和分支位置](Clip_2026-03-15_10-10-25.png)

看懂这两个位置以后，后面你就更容易判断自己当前是在本地工作区还是工作树，也更容易理解为什么右上角的按钮会跟着变化。

## 2. 创建新的工作树

**这一步要做什么：** 为新任务创建一个独立工作区。

你可以从两个入口进入，选一个最适合当前场景的即可。

### 方式 A：直接新建工作树

**适用场景：** 这是一个新任务，不需要继承之前对话的聊天历史。

**怎么操作：**

1. 点击左下角当前工作区位置。
2. 在菜单中选择“新工作树”。

**你会看到什么：** 底部菜单里会出现“新工作树”入口。

![从左下角菜单直接新建工作树](Clip_2026-03-15_10-13-17.png)

接下来继续选择工作树的起点：

**怎么操作：**

1. 保持环境为默认值即可，通常无环境也能正常开始。
2. 在右下角选择你要开始工作的分支。
3. 这组截图里只有 `main`，所以这里从 `main` 开始。

**你会看到什么：** 右下角会弹出分支选择器，当前分支会被标记出来。

![为新工作树选择起始分支](Clip_2026-03-15_10-14-45.png)

选好之后，像平时一样开始对话即可。

**你会看到什么：** Codex 会在开始对话时自动创建工作树，你可以在消息区看到创建日志。

![开始对话后自动创建工作树](Clip_2026-03-15_10-17-04.png)

### 方式 B：从已有对话派生到新工作树

**适用场景：** 你想保留之前某个对话的上下文，并把它派生到新的工作树里继续做。

**怎么操作：**

1. 在左侧找到你想继承历史的对话。
2. 右键该对话。
3. 选择“派生到新工作树”。

**你会看到什么：** 这个新会话会带着原来的聊天上下文，但工作目录已经是新的工作树。

![从已有对话派生到新工作树](Clip_2026-03-15_11-01-44.png)

不管你用哪种入口，目标都一样：先拿到一个独立工作树，再在这个工作树里继续让 Codex 工作。接下来就进入正式开发阶段。

## 3. 在工作树里继续工作，并创建工作分支

**这一步要做什么：** 让 Codex 在工作树里完成第一轮改动，并把改动挂到一个明确分支上。

如果你想额外确认工作树已经被创建出来，也可以在 VSCode 里看到对应分支和改动。这一步不是必须，只是辅助验证。

![可选：在 VSCode 中查看新工作树及其改动](Clip_2026-03-15_10-24-50.png)

**怎么操作：**

1. 继续在当前工作树会话里对话。
2. 当右上角出现“创建分支”时，点击它。

**你会看到什么：** Codex 会提示你把当前工作树上的改动放到一个正式分支里。

![在工作树中为当前改动创建分支](Clip_2026-03-15_10-19-43.png)

接着填写分支名：

**怎么操作：**

1. 可以使用系统默认生成的名称。
2. 也可以手动改名。
3. 这组截图里我把它命名为 `codex/a`。

**你会看到什么：** 弹窗里会出现默认分支名输入框，确认后当前工作树就会切到这个新分支。

![为工作树分支设置名称](Clip_2026-03-15_10-20-05.png)

到这里为止，工作树里的改动已经有了明确归属。下一步不是立刻合并，而是先把结果切到本地看一眼，确认它在你的真实运行环境里是不是符合预期。

## 4. 把工作树结果切到本地验证

**这一步要做什么：** 把工作树上的结果签出到本地工作区，方便你直接运行、调试和验证效果。

**怎么操作：**

1. 在右上角点击“移动到本地”。

**你会看到什么：** 当前工作树分支会准备签出到本地工作区。

![点击右上角的移动到本地](Clip_2026-03-15_10-22-00.png)

如果你的本地工作区不干净，Codex 会先拦住这次操作。

**你会看到什么：**

- 弹窗会提示你先处理本地改动
- 这时先在本地执行 `git stash` 或先提交，再回来继续

![本地工作区不干净时的提示](Clip_2026-03-15_10-22-25.png)

移动成功后，再回到当前对话确认界面状态。

**你会看到什么：**

- 左下角从工作树变成“本地”
- 右下角分支变成刚刚创建的 `codex/a`
- 右上角按钮从“移动到本地”变成“移至工作树”

![移动成功后界面状态发生变化](Clip_2026-03-15_10-37-04.png)

这一步是整个流程里最关键的切换点。到了这里，你已经从“让 Codex 在工作树里生成结果”，进入“我在本地亲自验证结果”的阶段了。

## 5. 决定继续在本地改，还是把会话移回工作树

**这一步要做什么：** 根据你对效果的判断，决定下一步在哪里继续工作。

现在你已经可以在本地直接查看结果，比如启动前后端、手动点功能、看页面效果，或者做你平时习惯的验证动作。

如果结果还需要调整，通常有两种选择：

- 小改动：继续留在本地对话，边改边看效果更直接
- 大改动：点击右上角“移至工作树”，让 Codex 回到工作树继续长时间修改，本地工作区则可以空出来做别的事

**你会看到什么：** 当会话切回工作树后，右上角仍然会用这个按钮提示当前可以来回切换。

![右上角的移至工作树按钮](Clip_2026-03-15_10-38-44.png)

注意：

- “移动到本地”和“移至工作树”这两个操作，在 Codex 正在思考或生成代码时通常不可用
- 所以你最好在它开始长时间运行之前，就先决定接下来想在哪个工作区继续

如果这时你已经确认结果没问题，就可以进入最后一个大阶段：把这次工作安全地并回目标分支。下面保留两种做法，你可以挑自己更顺手的一种。

## 6. 把改动合并回目标分支

下面假设你最终想把结果合并回 `main`。这里保留两种做法，你按自己的习惯选一种即可。

### 做法 A：把当前改动直接移到本地 `main`

**适用场景：** 你想快速把当前未提交改动带回本地 `main`，不强调保留独立分支历史。

**怎么操作：**

1. 在当前本地工作区先执行 `git stash`，把未提交改动放进暂存区。
2. 切换到本地 `main` 分支。
3. 再执行 `git stash pop`，把刚才的改动恢复出来。

**你会看到什么：**

- `git stash` 后，本地改动会先被收起来
- 切回 `main` 后，再 `pop` 就能把这些改动带到 `main`

![先把当前改动执行 git stash](image.png)

![切换回本地 main 分支](image-1.png)

![在 main 上执行 git stash pop 后恢复改动](Clip_2026-03-15_10-50-45.png)

### 做法 B：先在 `codex/a` 提交，再由 `main` 主动合并

**适用场景：** 你想保留工作树分支的提交历史，再把它规范地合并回 `main`。

第一步先让 `codex/a` 跟目标分支保持同步。

**怎么操作：**

1. 在 VSCode 或你习惯的 Git 图形化工具里，找到“拉取自”。
2. 选择目标分支，这里是 `main`。

**你会看到什么：**

- 菜单里需要选的是“拉取自”，不是普通“拉取”
- 接着会弹出可选来源分支列表

![在图形化工具中选择拉取自](Clip_2026-03-15_10-41-14.png)

![选择把 origin/main 作为同步来源](Clip_2026-03-15_10-41-39.png)

同步完成后，在 `codex/a` 上正常提交。

提交完成以后，你可以有两种回到主线的方式：

- 在 Codex Desktop 里点击“移至工作树”，让当前对话回到工作树
- 或者直接在 Git 工具里手动签出到 `main`

![在 Codex Desktop 中把会话移回工作树](image-2.png)

当你已经切回本地 `main` 后，再执行分支合并。

**怎么操作：**

1. 在分支菜单里选择“合并”。
2. 选择刚才提交过的 `codex/a`。
3. 把该分支的提交合并到 `main`。

**你会看到什么：** `main` 会吸收 `codex/a` 上的提交，而不是只拿到一份未提交文件。

![在分支菜单里找到合并入口](image-4.png)

![选择把 codex/a 合并到 main](image-5.png)

合并完成后，再确认一下结果。

**你会看到什么：** 提交历史里可以看到 `main` 已经包含了这次工作内容。

![检查 main 已经包含这次提交](Clip_2026-03-15_10-59-21.png)

最后正常同步到远程即可。

![完成后把改动同步到远程](Clip_2026-03-15_11-00-00.png)

到这里，代码已经回到了你的目标分支，工作树也完成了它的使命。最后只剩一个收尾动作：把不再需要的工作树删掉。

## 7. 删除不再需要的工作树

**这一步要做什么：** 在确认改动已经合并完成后，清理掉不再需要的工作树。

**怎么操作：**

1. 在 VSCode 中找到对应的工作树。
2. 右键你想删除的那个工作树。
3. 选择“删除工作树”。

**你会看到什么：** 菜单里会直接提供删除入口。

![在 VSCode 中删除不再需要的工作树](Clip_2026-03-15_11-04-46.png)

## 一句话回顾

这个工作流最重要的心智模型就是：先让 Codex 在工作树里独立做事，再按需要把结果切到本地验证，最后选择一种适合你的方式合并回主分支。
